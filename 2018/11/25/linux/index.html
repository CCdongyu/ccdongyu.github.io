记录一些关于Linux/Ubuntu的常见问题。

后台运行的几种方式

操作系统为Ubuntu 18.04.
在执行所有操作前，执行sudo su.
以0.10.2版本的kafka为例进行说明。

&

在Linux中，如果要让进程在后台运行，一般情况下，只需要在命令后面加上&即可。

这样操作的原理是将这个进程加入到一个作业队列里，通过jobs命令可以查看当前的作业队列。

ctrl+z bg

对于已经在shell前台中执行的程序，可以通过ctrl+z暂停程序，然后通过bg命令将停止的作业放入后台执行。

通过jobs命令可以看到,命令后面没有参数&

然后执行bg命令，将进程放入后台执行

再次通过jobs命令查看当前作业情况，可以发现命令后面多了个参数&，表示当前作业在后台运行。

对于想要返回到前台的作业，可以通过指定作业编号的方式返回前台

&和ctrl+z bg的问题

不管是通过&还是ctrl+z bg让进程在后台运行，所有作业的父进程都是当前shell终端，一旦父进程退出，所有子进程会受到hangup信号然后退出。如果想让子进程在父进程推出后仍然能够运行，则需要通过其他方式来达成。

setsid

通过man setsid，可以看到这样的描述runs a program in a new session.
大致的意思就是在一个新的会话里运行一个程序。

我们通过setsid命令运行进程

执行之后，进程自动在后台运行，通过jobs命令，发现当前shell下无作业，我们在通过ps -ef |grep zoo命令查找刚刚我们启动的进程,发现他被挂在到进程id为1671的父进程下执行了

再通过ps -ef |grep 1671，可以得知进程id为1671的进程为lib/systemd/systemd --user,并且还有很多的其他进程也作为这个进程的子进程在运行着。

这样的话，即使退出当前shell终端，进程也不会结束，而是继续在进程id为1671的父进程下继续执行

nohup

通过man nohup，可以看到这样的描述Run COMMAND, ignoring hangup signals.
大致意思是，运行命令，忽略hangup信号。

通过nohup命令运行程序

通过终端上的反馈nohup: ignoring input and appending output to 'nohup.out'可以知道，通过这种方式运行会忽视输入，并把输出重定向到了nohup.out文件里。同时通过jobs命令我们也可以看到刚才运行的进程在当前shell终端下，父进程仍然为当前终端。

当我们关闭当前shell终端再重新打开一个shell终端时，通过ps -ef|grep zoo命令查找我们发现，该进程仍在运行，并且父进程变为了进程id为1671的systemd进程。
